### 2026/06/10: リポジトリ構造の刷新，GPU推論の最適化，DeepSeek-R1思考ログ基盤の構築

- **実施したこと**:
  - **大規模なディレクトリ再編**: リポジトリを `code/` (コアロジック), `code/scripts/` (実行スクリプト), `results/` (集約された出力) の3層構造に刷新．`hcp` と `llm` のパッケージ化により保守性を向上．
  - **出力管理の高度化**: ベンチマーク結果を `results/benchmarks/<モデル>/<アルゴリズム>/<手法>/run_<日時>` の階層構造で保存するように刷新．大規模な実験データの整理と管理を容易にした．
  - **Program-of-Thought (PoT) の実装**: LLMに Python コードを書かせ，それをローカル環境で自動実行する `CodeExecutor` を開発．AIの「暗算ミス」を物理的に排除する評価手法を確立．
  - **秘密のテーブル（鍵）の公開機能**: HCPの核心である `sgm` テーブルをプロンプトに埋め込む機能を実装．AIに「鍵」を与えた状態での暗号解読シミュレーションを可能にした．
  - **ローカルLLM環境のGPU化**: Nix環境からホストOS側の Ollama へ移行し，RTX 2080 Ti をフル活用．Gemma 2 (9b) などの大型モデルを実用的な速度で回せる環境を構築．
  - **パースロジックの究極の安定化**: 16kトークン窓の確保，ストリーミング受信，思考タグ内からのフォールバック抽出を組み合わせ，パースエラーをほぼゼロに抑え込んだ．

- **得られた知見**:
  - **「推論」と「実行」の分離**: `simple_add` において，ヒントなし（Pure）では 5% だった正解率が，ヒント＋コード実行（Rationale_PoT）では **100%** に跳ね上がった．これは，AIが「自力でルールを見つけるのは苦手だが，教えられたルールを正確に実行するのは得意」であることを鮮明に示している．
  - **HCPの強固さの源泉**: `func_31` において，秘密の鍵（sgm）と計算手順の両方を与えても，AIは Python コード化の段階で変数名を間違えるなどのハルシネーションを起こし，正解できなかった．これは HCP が「AIのワーキングメモリ（文脈保持能力）の限界」を突く有効な防御策であることを示唆している．
  - **モデルによる特性差**: DeepSeek-R1 は深く考えすぎて自爆（トークン制限）しやすく，Gemma 2 は論理的にヒントに従う素直さがあるなど，モデルごとの「思考のクセ」が HCP 解読において重要な変数となることを確認した．

- **次にやること（Next Actions）**:
  - 構築した「パラダイム比較基盤（`run_paradigms.py`）」を用い，各モデル（Qwen, DeepSeek, Gemma）の得意・不得意を全アルゴリズムで網羅的にデータ収集する．
  - AIが「鍵」をどのように参照して間違えたか，`reasoning_logs/` の定性分析を行い，HCP のどのステップが AI にとって最も困難かを特定する．
  - さらなる高精度化のため，AIに「一度書いたコードを自分でテストさせる」といった Iterative Refinement の導入を検討する．
  - **ベンチマーク運用ガイドの作成**: 誰でも（将来の自分も）実験を再現できるよう `BENCHMARK_GUIDE.md` を作成．

- **得られた知見**:
  - **推論モデルの特性**: DeepSeek-R1 は 1 問に 4000 トークン以上を費やすことがあり，従来の LLM 設定（256〜512制限）では回答に到達する前に打ち切られてしまう．この「長考」を遮らない設定が HCP 解読には不可欠である．
  - **Jagged Frontier の片鱗**: `func_31` において DeepSeek-R1 (7b) が正解を出すケースを確認．思考ログからは，数列全体を俯瞰して特定の定数に注目しようとする「ひらめき」に近い挙動が見られ，質的分析の重要性が浮き彫りになった．
  - **Nix と GPU ドライバの不整合**: 再現性重視の Nix 環境でも，ホスト OS の最新ドライバとの動的リンクが切れることがあり，物理レイヤー（ホスト OS）での Ollama 運用が研究効率と性能の両面で現実解となる．

- **次にやること（Next Actions）**:
  - DeepSeek-R1 (7b) を用い，各アルゴリズム（特に `simple_add` と `func_31`）に対し各 20 件以上の大規模ベンチマークを実行し，統計データと「思考の軌跡」を収集する．
  - 収集した `reasoning_logs/` を読み込み，LLM がどのような論理の飛躍や誤解（ハルシネーション）を経て結論に至っているかを定性的に分類・整理する．
  - 正解率向上のため，「数列の順序が重要である」といったメタなヒントをプロンプトに含めた際の挙動変化を調査する．

- **実施したこと**:
  - APIキーの設定と疎通確認：Google AI StudioからAPIキーを取得し，`~/.env` へ設定した．
  - モデルの `max_output_tokens` 拡張および `thinking_budget` の制御：`gemini-2.5-flash` の思考（Thinking）プロセスによるトークン切れとオウム返しを防ぐため，`max_output_tokens` を `8192` に広げ，`thinking_config = types.ThinkingConfig(thinking_budget=1024)` を設定して思考プロセスを抑制した．
  - レスポンスパーサーの堅牢化：Markdown記号の除去，全角コロン（`Answer：5`），日本語表現（`答え: 5`），`Z = 5` の単語境界マッチ，最終行の単一数値抽出，末尾数値抽出の5段階のフォールバックを実装し，`ERR` の発生を劇的に減少させた．

  - AI Studioの無料枠プロジェクトではモデルごとに1日のリクエスト制限（20回程度）があるため，実験の規模に合わせたプロジェクト設計が必要である．

### 2026/06/08: LLMベンチマークスクリプト（`code/llm_benchmark/`）の実装

- **実施したこと**:
  - 研究計画書の「評価軸1: Few-shotインコンテキストルール推論」を実装するため，`code/llm_benchmark/` パッケージを新規に作成した．
  - **`data_generator.py`**: 既存の `ComputablePasswordGenerator` をラップし，Few-shot 例題用とテスト用の DataFrame を生成する関数を実装した．ジェネレータの選択は `AVAILABLE_GENERATORS` dict で管理し，新ジェネレータの追加が容易な設計にした．
  - **`prompt_builder.py`**: `PromptBuilder` 抽象基底クラスと `TextPromptBuilder` を実装した．将来の画像入力（マルチモーダル）への拡張を意識したポリモーフィックな設計を採用した．
  - **`gemini_client.py`**: `google-genai` SDK（最新版, v1.67.0）を使用した Gemini API クライアントを実装した．`temperature=0.0` による決定論的出力の促進，正規表現によるレスポンスのパース（`Answer: <数字>` 形式），レートリミット対応のスリープ処理（デフォルト 4秒/リクエスト）を実装した．
  - **`evaluator.py`**: `BenchmarkRecord` と `Evaluator` クラスを実装し，正解率の計算，結果詳細の CSV 保存，実験メタデータの JSON 保存，コンソールサマリー出力を実装した．
  - **`benchmark_runner.py`**: `argparse` を用いた CLI エントリーポイントを実装した（`--generator`, `--n_shot`, `--n_test`, `--model`, `--sleep_sec`, `--seed` など）．
  - **`dry_run_test.py`**: API キーなしで全モジュールの動作を検証するドライランテストスクリプトを作成し，正常動作を確認した．
  - `flake.nix` と `requirements.txt` に `google-genai` パッケージを追加した．
  - 全実装をコミット（`634b2b7`）した．

- **得られた知見**:
  - `google-generativeai` パッケージは非推奨となっており，後継の `google-genai` SDK（`from google import genai` でインポート）への移行が必要であることを確認した．
  - Nix flake 環境での Python パッケージ追加は `flake.nix` の `withPackages` に追記するだけで完結し，`nix develop` 実行時にキャッシュから取得・環境再ビルドが行われる．
  - ドライランテストにより，データ生成 → プロンプト構築 → 評価器の全パイプラインが正常に動作することを API 呼び出しなしで検証できた．

- **次にやること（Next Actions）**:
