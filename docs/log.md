### 2026/07/01: PoT学習方針の確定、Google Drive同期体制の確立、パースエラー問題の発見と対応

- **実施したこと**:
  - **Google Drive 同期体制の確立（rclone）**: 実験結果の長期保存先を OneDrive から Google Drive に移行。`rclone` を用いて `results/` フォルダを `gdrive:human-computable-passwords-results` に対し双方向同期する設定を確立。当初は `rclone mount` によるFUSEマウント方式を採用したが、ダッシュボードの読み込み速度（Loadingが長い問題）が生じたため、マウント方式を廃止しローカルフォルダのまま運用する方針に変更。実験終了後に `rclone sync` を自動実行するフック（`run_prompting.py` 内のバックグラウンドサブプロセス）を導入し、「実験完了ごとに自動同期」方式とした。
  - **パースエラー問題の発見（パースによる偽陰性の確認）**: `qwen2.5_0.5b_ft` の評価ログ（Stage 0 / `simple_add`）を確認したところ、モデルが JSON フォーマット（`{"answer": X}`）の代わりに日本語自然言語で「出力は X を出力します」「差分の値が XX なので〜」と回答してしまい、**パーサーが ERR（正解なしと判定）を返していた**ことを発見。これは「モデルが理解できていない」のではなく「モデルの出力フォーマットを現行パーサーが読み取れない」という**評価パイプライン側の問題（偽陰性）**であることを認識。
  - **PoT（Program-of-Thought）学習方針の確定**: パースエラー問題への根本的解決策として、学習ターゲット（Completion）を JSON から **Python コードブロック（`def predict_z(X): ...`）**に切り替える PoT 方式を採用することを決定。この方針は以下の理由で既存研究との比較上も妥当である：
    - 従来のML（CNN/Bi-LSTM等）が「重みの中で鍵とルールを同時に学習（暗記＋演算の一体化）」するのに対し、LLMは**「重みの中に鍵テーブルを暗記し、演算はPythonコードとして外部化（暗記と演算の分離）」**するという、LLM特有の役割分担の妥当性を測る新規の実験軸（RQ3）として位置づけられる。
    - パースエラーによるノイズを排除し、「モデルが生成したコードを実行した結果」により純粋な帰納・演繹能力の評価が可能になる。
  - **PoT 学習パイプラインの実装**: `train_finetuning.py` に `--use_code` フラグを追加。`generate_target_code()` ヘルパー関数により、各ジェネレータ（`simple_add`, `func_13`, `func_22`, `func_31`, `func_pow`）に対応した教師コードを自動生成する機能を実装。
  - **動作確認（PoT 評価パイプラインの疎通確認）**: `Qwen2.5-3B-Instruct` を `func_22 / stage0` に対し 10 件で PoT 学習（`--use_code`）を行い、その後 `--paradigm pot` で評価を実行。モデルが正しく `def predict_z(X): ...` のコードブロックを生成し、それが内部実行されて数値予測に変換されるパイプライン全体の疎通を確認した。
  - **CPU/メモリ最適化の適用**: `SFTConfig` に `dataloader_num_workers=4`, `dataloader_pin_memory=True` を追加し、データローダーのマルチコアCPU並列化とCPU→VRAM転送の高速化を実施。

- **得られた知見**:
  - **「パースエラーは評価の偽陰性である」という認識の確立**: モデルがJSON以外の形式で回答した場合に即座に ERR と判定するパーサーは、実際の正解率を過小評価する危険がある。PoT に移行することで、出力フォーマットがコードブロックに固定され、抽出・実行が機械的かつ確実になる。
  - **評価時の生成速度と学習品質の関係**: 未学習または学習不足のモデルは、指示に従えず Few-shot例題等を日本語で延々と繰り返し出力してしまう（バブリング）。これにより毎回の生成がトークン上限（512）まで走り評価が極端に遅くなる。十分に学習されたモデルは、コードブロックだけをピンポイントで出力して即 EOS を吐くため、評価速度が 10〜15 倍以上改善される見込み。
  - **Stage 0 × PoT の学術的意義**: Stage 0（プロンプトに鍵もルールも明示しない）でも、PoT 学習ではモデルが教師データ（Completion）として鍵とルールが書かれたコードを「見て」学習する。つまり、「プロンプト上は完全ブラインド」「重みの中に鍵を刷り込む」という構図になっており、これはケルクホフスの原理（アルゴリズム公開・鍵秘匿）のシナリオを最も正確にシミュレートしている。
  - **なぜ従来MLはアルゴリズムを学習できてLLMファインチューニングは難しいのか（理論的考察）**:
    - **勾配の到達範囲の違い**: CNN/Bi-LSTMは、損失（正解との誤差）の勾配が入力Xから出力Zまでの**全変換ステップ**に逆伝播する。一方、LLMのLoRAファインチューニングでは損失は「出力トークン列（例：`{"answer": 2}`）の予測誤差」にしかかからず、入力X（プロンプト側）への勾配はほとんど届かない。つまり「XからZへの変換方法」を学ぶ勾配信号がほぼ存在しない。
    - **LoRAの表現力制約（更新パラメータ量の問題）**: LoRAは全パラメータの約1%にしか更新をかけない低ランク近似更新である。鍵テーブル（26値）の記憶と演算回路（mod、間接参照）の両方を同時にこの小さなアダプター層に刻み込むには表現力が根本的に不足する可能性がある。CNNは全重みがこの1タスクに専念できる。
    - **ショートカット学習のリスク**: CNNにとっての正解ラベル「Z=2」は入力Xとの変換関係を学ぶ直接信号だが、LLMにとっての`{"answer": 2}`は「このプロンプト形式の後にこのトークンが来やすい」という表面的パターンとして学習されうる。アルゴリズムを理解した上で正解を導いているのか、統計的ショートカットを学んだのかが区別できない。
    - **PoT がこの問題を解消する理由**: 教師ターゲットを `def predict_z(X): sgm=[...]; ...` というコード列にすると、コードの全トークン（鍵の数値、演算子、インデックス）の予測に損失がかかる。これにより「鍵テーブルの各値の記憶」と「計算ロジックの構造」の両方に豊かな勾配信号が流れるため、LoRAアダプターへの情報量が大幅に増加する。

- **次にやること（Next Actions）**:
  - `func_22 / stage0` に対し、本番スケールの PoT 学習（`--n_train 800 --n_val 200 --epochs 3`）を実施し、PoT 学習済みモデルの汎化精度（100件テスト）を測定する。
  - 比較実験として、同条件で `--use_code` なし（Pure/JSON）の学習を実行し、PoT vs Pure の精度差を定量化する。
  - 複数のアルゴリズム（`simple_add`, `func_31`, `func_22`）と複数のモデルサイズ（0.5B, 1.5B, 3B）の組み合わせで網羅的なベンチマークを実施する（第2期の準備）。


### 2026/06/29: LLMファインチューニング環境（Nix+pip）の確立、QLoRA学習・評価パイプラインの構築

- **実施したこと**:
  - **環境管理方式の整理（Nix & pip）**: NixOS環境で PyTorch 等の大規模パッケージをソースコンパイルすると発生するOSフリーズの問題に対処するため、Nixで動的ライブラリ（CUDAドライバパス、libstdc++、zlib）のパスを解決しつつ、Pythonパッケージは仮想環境内の pip 経由でプリコンパイル済みバイナリ（Wheels）を直接取得する「Nixシステム管理＋Pipライブラリ管理」のハイブリッド環境を確立。`requirements.txt` に PyTorch、transformers、peft、bitsandbytes、trl、accelerate、datasets を明確に列挙。
  - **遅延インポート導入によるCUDA競合回避**: `core/__init__.py` において、TensorFlowを依存に持つ `Models` および `Utils` モジュールをPythonの `__getattr__` で遅延ロード（Lazy Import）するようにリファクタリング。これにより、LLMの学習を実行する際に TensorFlow のCUDAコンテキストの初期化が完全にバイパスされ、PyTorchによるGPU（RTX 2080 SUPER）での学習が競合・クラッシュすることなく安全に行えるようになった。
  - **ファインチューニングスクリプト (`train_finetuning.py`) の実装**: QLoRA (4-bit量子化LoRA) を用い、指定したアルゴリズム・開示ステージに応じて SFTTrainer (TRL v1.7.0準拠) でLLMを微調整する学習スクリプトを作成。
  - **学習済みモデルの保存先整理**: ファインチューニングの結果もプロンプティングと同様に整理されるように、`results/finetuned_models/{model}/stage{stage}/{generator}/run_{timestamp}` の階層構造に出力先を再編成。学習済み LoRA アダプターが復元可能な形で保存される。
  - **評価スクリプトのベンチマーク統合**: 最初は単独の `eval_lora.py` として構築したが、ベンチマーク共通スクリプトである `run_prompting.py` に `lora` プロバイダを追加する形で完全に評価ロジックを統合した。これにより、ファインチューニングモデルの評価時にも並列処理、結果CSV、連番思考プロセスログ出力（`reasoning_logs/`）などの強力な評価基盤がすべて共有されるようになり、不要となった `eval_lora.py` は削除した。評価結果は `results/evals/` 配下に歴代モデルとともに整理される。
  - **0.5Bモデルによる動作検証（ドライラン）の完走**: `Qwen/Qwen2.5-0.5B-Instruct` モデルを用い、HCPの `simple_add`（Stage 1）データで学習＆統合後のベンチマーク評価を実行し、正常にトレーニングおよび評価が10秒台で完了することを確認。

- **得られた知見**:
  - NixOS上でのディープラーニング環境構築において、`nix-ld` と `shellHook` で `stdenv.cc.cc.lib` のパスを通すことで、pipのバイナリパッケージをそのまま安定して稼働させられる。
  - 同一Pythonプロセス内で TensorFlow と PyTorch の両方を読み込むと、CUDAドライバのロック競合（Error 302: Error loading CUDA libraries）が発生する。インポート順序の制御や、モジュールの遅延読み込み（Lazy Loading）が競合防止に非常に有効である。

- **次にやること（Next Actions）**:
  - 軽量LLM（Qwen2.5-1.5B 等）を実際に `n_train = 500`〜`1000` 程度のデータサイズ、数エポックでファインチューニングし、プロンプティング単体（インコンテキスト推論）と比較した際、どの程度ルールや秘密鍵の構造を重みに焼き付けて暗記・演算できるようになるかを検証する。

### 2026/06/10: リポジトリ構造の刷新，GPU推論の最適化，DeepSeek-R1思考ログ基盤の構築

- **実施したこと**:
  - **大規模なディレクトリ再編**: リポジトリを `code/` (コアロジック), `code/scripts/` (実行スクリプト), `results/` (集約された出力) の3層構造に刷新．`hcp` と `llm` のパッケージ化により保守性を向上．
  - **出力管理の高度化**: ベンチマーク結果を `results/benchmarks/<モデル>/<アルゴリズム>/<手法>/run_<日時>` の階層構造で保存するように刷新．大規模な実験データの整理と管理を容易にした．
  - **Program-of-Thought (PoT) の実装**: LLMに Python コードを書かせ，それをローカル環境で自動実行する `CodeExecutor` を開発．AIの「暗算ミス」を物理的に排除する評価手法を確立．
  - **秘密のテーブル（鍵）の公開機能**: HCPの核心である `sgm` テーブルをプロンプトに埋め込む機能を実装．AIに「鍵」を与えた状態での暗号解読シミュレーションを可能にした．
  - **ローカルLLM環境のGPU化**: Nix環境からホストOS側の Ollama へ移行し，RTX 2080 Ti をフル活用．Gemma 2 (9b) などの大型モデルを実用的な速度で回せる環境を構築．
  - **パースロジックの究極の安定化**: 16kトークン窓の確保，ストリーミング受信，思考タグ内からのフォールバック抽出を組み合わせ，パースエラーをほぼゼロに抑え込んだ．

- **得られた知見**:
  - **「推論」と「実行」の分離**: `simple_add` において，ヒントなし（Pure）では 5% だった正解率が，ヒント＋コード実行（Rationale_PoT）では **100%** に跳ね上がった．これは，AIが「自力でルールを見つけるのは苦手だが，教えられたルールを正確に実行するのは得意」であることを鮮明に示している．
  - **HCPの強固さの源泉**: `func_31` において，秘密の鍵（sgm）と計算手順の両方を与えても，AIは Python コード化の段階で変数名を間違えるなどのハルシネーションを起こし，正解できなかった．これは HCP が「AIのワーキングメモリ（文脈保持能力）の限界」を突く有効な防御策であることを示唆している．
  - **モデルによる特性差**: DeepSeek-R1 は深く考えすぎて自爆（トークン制限）しやすく，Gemma 2 は論理的にヒントに従う素直さがあるなど，モデルごとの「思考のクセ」が HCP 解読において重要な変数となることを確認した．

  - クォータ制限（429エラー）および混雑（503エラー）に対する自動リトライロジックを `GeminiClient.predict` 内に実装：エラーに含まれる推奨待機秒数を動的に解析し，自動待機（指数バックオフ）して再試行する機構を組み込んだ．
  - 無料枠の厳しいリクエスト数制限（1日20リクエスト）に伴い，新しいGCPプロジェクトを立ち上げて新キーを再生成し，登録した．
- **得られた知見**:
  - `gemini-2.5-flash` のような思考モデルはそのままでは推論時にFew-shot例題のリストを丸ごとオウム返しするループに陥りやすく，`thinking_budget` を1024等に制限することがトークン切れと応答フォーマット破損の防止に極めて有効である．
  - AI Studioの無料枠プロジェクトではモデルごとに1日のリクエスト制限（20回程度）があるため，実験の規模に合わせたプロジェクト設計が必要である．
